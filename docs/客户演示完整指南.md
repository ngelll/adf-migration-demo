## 客户现场演示流程(15-20 分钟)

### Act 1 - 让客户看懂"一份代码"(2 分钟)

> "这是 GitHub 上的仓库,**只有一份 JSON 配置**。"

操作:
1. 浏览器打开 `https://github.com/ngelll/adf-migration-demo`
2. 点 `factory/` 文件夹 → 展示 2 个 ADF 共用一套结构,但 GlobalParameter 不同
3. 点 `linkedService/AzureBlobStorageLS.json` → 展示 `@{linkedService().StorageEndpoint}` 参数化

### Act 2 - VS Code 里看本地副本(2 分钟)

> "我已经 clone 到本地了。文件树跟 GitHub 一模一样。"

操作:
1. 切到 VS Code
2. 点开左侧 Explorer,展示文件树
3. 双击 `factory/adf-src-eastasia-demo.json` 和 `factory/adf-dst-southeastasia-demo.json` 对比 GlobalParameter

### Act 3 - 用脚本看运行态(3 分钟,**最有冲击力**)

> "现在我跑一个脚本,**对比 Git 上的代码和 Azure 上的真实运行态**。"

操作:
1. VS Code 终端(Ctrl + ` 打开)
2. 切到 PowerShell tab
3. 跑:
   ```powershell
   cd tools
   .\compare-dev-vs-runtime.ps1 -Factory adf-src-eastasia-demo
   ```
4. **重点解说输出**:
   - `[开发态/Git main]: StorageEndpoint = https://stadfdemobe002d.blob.core.windows.net`(EA)
   - `[运行态/工厂]: StorageEndpoint = https://stadfdemobe002d.blob.core.windows.net`(EA)
   - `✅ 开发态 == 运行态`
5. 再跑目标 ADF:
   ```powershell
   .\compare-dev-vs-runtime.ps1 -Factory adf-dst-southeastasia-demo
   ```
6. **关键对比**:
   - 这两个 ADF 用**完全相同的 JSON 代码**(linkedService/dataset/pipeline)
   - 只有 GlobalParameter 不同 → 各自连自己区域的 Storage

### Act 4 - 触发管道(3 分钟)

```powershell
$runId = az datafactory pipeline create-run `
  -g rg-adf-migration-demo `
  --factory-name adf-dst-southeastasia-demo `
  --name MigrationDemoPipeline `
  --query runId -o tsv
echo "Run ID: $runId"

# 等 30 秒
Start-Sleep -Seconds 30

# 看状态
az datafactory pipeline-run show `
  -g rg-adf-migration-demo `
  --factory-name adf-dst-southeastasia-demo `
  --run-id $runId `
  --query "{status:status, duration_ms:durationInMs}" -o table
```

### Act 5 - 看真实数据(2 分钟,**收尾爆点**)

```powershell
# 下载源 ADF 运行后的输出(EA Storage)
az storage blob download `
  --account-name stadfdemobe002d `
  --auth-mode login `
  -c destination-data -n sample_copy.csv -f $env:TEMP\ea_result.csv
Get-Content $env:TEMP\ea_result.csv
# 应该看到 alice/bob/charlie/david/eve(EA 数据)

# 下载目标 ADF 运行后的输出(SEA Storage)
az storage blob download `
  --account-name stadfdemosea7660 `
  --auth-mode login `
  -c destination-data -n sample_copy.csv -f $env:TEMP\sea_result.csv
Get-Content $env:TEMP\sea_result.csv
# 应该看到 frank/grace/henry/iris/jack(SEA 数据)
```

> "看,**同一份 JSON,两个 ADF,数据完全分离 —— EA 是 alice/bob,SEA 是 frank/grace,谁也不串**。这就是 GitOps 多区域的核心价值。"

### Act 6 - 想加新区域怎么办?(2 分钟)

> "假如客户明年要扩到美国区域,需要做的事:"

操作:打开 VS Code,新建一个文件 `factory/adf-us-eastus2-demo.json`,展示只需要改一行 GlobalParameter 指美国 Storage 就行。

```json
{
  "name": "adf-us-eastus2-demo",
  "properties": {
    "globalParameters": {
      "StorageEndpoint": {
        "type": "String",
        "value": "https://stadfdemoeastus2.blob.core.windows.net"  
      }
    }
  }
}
```

> "其他 LinkedService/Dataset/Pipeline 一行都不用改。这就是企业 dev/staging/prod 多环境的标准做法。"

## 演示中可能被问的 5 个问题(预案)

**Q1: 这跟 ARM 模板部署有什么区别?**

A: ARM 模板是 "全量替换" 思路,Git 是 "细颗粒度版本管理"。Git 模式让你可以 review 每一行 JSON 改动,可回滚,可分支开发,可 CI/CD。ARM 更适合一次性部署。

**Q2: 多人协作怎么办?**

A: 用 Git 的 feature branch + Pull Request,跟开发代码一样。ADF Studio 支持 feature branch 切换,每个人在自己分支开发,merge 到 main 之后再统一发布。

**Q3: 敏感信息(密码/connection string)怎么办?**

A: 用 Azure Key Vault。LinkedService 里只存 "去哪个 Key Vault 取哪个 Secret",真实凭据永远不进 Git。

**Q4: 怎么自动化?**

A: GitHub Actions(或 Azure DevOps Pipeline)监听 `adf_publish` 分支变化,自动跑 `az deployment group create` 部署 ARM 到所有目标 ADF。一次提交,自动同步到所有区域。

**Q5: 真实生产里数据迁移怎么办?**

A: 用 `azcopy` 做一次性历史数据搬运 + 这套 ADF 管道做增量同步。或者起一个迁移专用 ADF 管道,跨区域 Copy Activity 自带断点续传。
